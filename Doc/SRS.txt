# Tài liệu Đặc tả Yêu cầu Hệ thống (SRS)
## Dự án: Ứng dụng Bán lẻ Thời trang Big Size (Big Size Fashion App)
### Phiên bản: 2.0 | Ngày cập nhật: 27/02/2026

---

## 1. Mở đầu (Introduction)

### 1.1 Mục đích (Purpose)
Tài liệu này đặc tả toàn bộ yêu cầu chức năng, phi chức năng và kỹ thuật cho dự án "Ứng dụng Bán lẻ Thời trang Big Size". Hệ thống được thiết kế đặc biệt nhằm giải quyết "nỗi đau" của người dùng có ngoại cỡ (big size) trong việc tìm kiếm, chọn lựa trang phục và kích cỡ phù hợp thông qua **tư vấn AI** và **quản lý hồ sơ số đo cá nhân**.

### 1.2 Phạm vi dự án (Scope)
Hệ thống bao gồm:
- **Ứng dụng Mobile** (Android) cho Customer và Shipper.
- **Backend API** (ASP.NET Core) phục vụ toàn bộ nghiệp vụ.
- **Admin Dashboard** (Web) cho Staff và Admin quản lý hệ thống.

Các tính năng kỹ thuật đặc biệt bắt buộc:
1. **Map Tracking** — Theo dõi vị trí Shipper giao hàng real-time trên bản đồ.
2. **Thanh toán Online (SePay)** — Tích hợp cổng thanh toán SePay bên cạnh COD.
3. **Bảo mật OTP** — Gửi mã OTP qua Email khi đăng ký tài khoản và đổi mật khẩu.
4. **Chat AI** — Dùng AI tư vấn chọn sản phẩm và size phù hợp với số đo người dùng.

### 1.3 Thuật ngữ & Từ viết tắt
| Thuật ngữ | Giải thích |
|---|---|
| Big Size | Quần áo cho người có kích thước lớn, thường từ size XL trở lên |
| Body Profile | Hồ sơ số đo cơ thể (chiều cao, cân nặng, vòng 1/2/3,...) |
| Fit-Check | Kiểm tra độ vừa vặn giữa số đo người dùng và thông số sản phẩm |
| Size Guide | Bảng thông số kích thước chi tiết của từng size trong một sản phẩm |
| Variant | Biến thể sản phẩm theo Size + Màu sắc |
| SKU | Stock Keeping Unit — Mã định danh duy nhất cho mỗi biến thể |
| OTP | One-Time Password — Mã xác thực dùng một lần |
| SePay | Cổng thanh toán trực tuyến được tích hợp trong hệ thống |
| COD | Cash On Delivery — Thanh toán khi nhận hàng |
| JWT | JSON Web Token — Phương thức xác thực API |
| FCM | Firebase Cloud Messaging — Dịch vụ gửi Push Notification |
| Soft Delete | Xóa mềm — đánh dấu IsDeleted thay vì xóa khỏi database |

### 1.4 Nhóm người dùng (User Roles)
Hệ thống phân chia quyền truy cập dựa trên **4 vai trò**:

| Role | Giá trị | Mô tả |
|---|:---:|---|
| **Admin** | 1 | Quản trị viên — quyền cao nhất, quản lý toàn bộ hệ thống, dữ liệu nền tảng, nhân sự và báo cáo. |
| **Staff** | 2 | Nhân viên — quản lý sản phẩm, xử lý đơn hàng, quản lý voucher. |
| **Customer** | 3 | Khách hàng — người dùng cuối, mua sắm, đánh giá, sử dụng AI tư vấn size. |
| **Shipper** | 4 | Người giao hàng — nhận đơn, cập nhật vị trí GPS real-time, xác nhận giao hàng. |

## 2. Tổng quan Hệ thống (System Overview)

### 2.1 Kiến trúc hệ thống (System Architecture)
Hệ thống áp dụng mô hình **Clean Architecture** gồm 3 tầng:

```
┌─────────────────────────────────────────────────────┐
│              MV.PresentationLayer                   │
│         (Controllers, Middleware, Filters)           │
├─────────────────────────────────────────────────────┤
│              MV.ApplicationLayer                    │
│         (Services, Interfaces, DTOs, Validators)    │
├─────────────────────────────────────────────────────┤
│                MV.DomainLayer                       │
│         (Entities, Enums, Configurations)           │
├─────────────────────────────────────────────────────┤
│                  PostgreSQL                         │
│              (20 Bảng dữ liệu)                     │
└─────────────────────────────────────────────────────┘
```

### 2.2 Công nghệ sử dụng (Technology Stack)
| Thành phần | Công nghệ |
|---|---|
| Backend API | ASP.NET Core 8 (C#) |
| Database | PostgreSQL 15+ |
| ORM | Entity Framework Core |
| Authentication | JWT (Access Token + Refresh Token) |
| Mã hóa mật khẩu | BCrypt |
| Thanh toán | SePay API |
| Lưu trữ ảnh | Cloudinary |
| Push Notification | Firebase Cloud Messaging (FCM) |
| AI Chat | Tích hợp LLM API (Gemini / OpenAI) |
| Map Tracking | Google Maps SDK (Mobile) |
| Gửi Email OTP | SMTP (Gmail / SendGrid) |

### 2.3 Tổng quan Database (20 Bảng)

**Nhóm Người dùng & Xác thực (5 bảng):**
Users, UserAddresses, UserBodyProfiles, OtpCodes, RefreshTokens

**Nhóm Sản phẩm (5 bảng):**
Categories, Products, ProductVariants, SizeGuides, ProductImages

**Nhóm Giao dịch (5 bảng):**
CartItems, Wishlists, Vouchers, Orders, OrderItems

**Nhóm Thanh toán & Vận chuyển (2 bảng):**
Payments, ShipperLocations

**Nhóm Tương tác & Tiện ích (3 bảng):**
ProductReviews, ChatAiHistory, Notifications

**Luồng trạng thái đơn hàng:**
```
PENDING → CONFIRMED → PROCESSING → SHIPPING → DELIVERED

CANCELLED  CANCELLED   CANCELLED   CANCELLED (nếu giao thất bại 3 lần)
```

### 4.3 Quy trình Bảo mật OTP (OTP Verification Flow)

```
[Đăng ký tài khoản]
Customer nhập Email + Password
   → Hệ thống tạo User (IsEmailVerified = false)
   → Sinh OTP 6 số, lưu OtpCodes (Type = VERIFY_EMAIL, ExpiredAt = +5 phút)
   → Gửi OTP đến Email qua SMTP
   → Customer nhập OTP
   → Hệ thống kiểm tra: Code khớp, chưa hết hạn, IsUsed = false
   → Thành công: IsEmailVerified = true, IsUsed = true
   → Thất bại: Cho phép gửi lại OTP (tối đa 3 lần)

[Đổi mật khẩu]
Customer nhập mật khẩu cũ (verify BCrypt)
   → Hệ thống sinh OTP, gửi Email (Type = RESET_PASSWORD)
   → Customer verify OTP
   → Cho phép nhập mật khẩu mới
   → Hash BCrypt → cập nhật Users.Password
```

### 4.4 Quy trình Thanh toán SePay (SePay Payment Flow)

```
Customer chọn thanh toán SePay
   → Backend gọi SePay API tạo giao dịch (Amount, OrderCode)
   → SePay trả về URL thanh toán / QR Code
   → Customer hoàn tất thanh toán trên SePay
   → SePay gọi Webhook callback về Backend
   → Backend verify checksum + Amount
   → Cập nhật Payments: Status = COMPLETED, TransactionId, PaidAt
   → Gửi Notification cho Customer: "Thanh toán thành công"

[Thất bại / Hết hạn]
   → SePay callback: Status = FAILED
   → Backend cập nhật Payments.Status = FAILED
   → Gửi Notification cho Customer: "Thanh toán thất bại"

[Hoàn tiền khi hủy đơn]
   → Đơn hàng đã thanh toán SePay bị hủy
   → Backend cập nhật Payments.Status = REFUND_PENDING
   → Staff xử lý hoàn tiền thủ công hoặc qua SePay API
```

## 5. Yêu cầu Phi chức năng (Non-Functional Requirements)

### 5.1 Hiệu năng (Performance)
- Thời gian response API trung bình < 500ms.
- Thuật toán gợi ý size AI phản hồi < 1.0 giây.
- Cập nhật GPS shipper real-time với độ trễ < 3 giây.
- Database được tối ưu Index trên các bảng truy vấn thường xuyên (Products, Orders, ProductVariants).

### 5.2 Bảo mật (Security)
- Mật khẩu mã hóa **BCrypt** (cost factor ≥ 11).
- Xác thực API bằng **JWT** (Access Token: 15 phút, Refresh Token: 7 ngày).
- OTP có thời hạn **5 phút**, tối đa **3 lần** gửi lại mỗi email.
- Webhook SePay bắt buộc **verify checksum** để chống giả mạo.
- Tất cả API bảo vệ bằng **Authorization middleware** phân quyền theo Role.
- Chống SQL Injection qua Entity Framework Core (parameterized queries).

### 5.3 Khả năng mở rộng (Scalability)
- Database PostgreSQL với cấu trúc rõ ràng, phân biệt bảng tĩnh (Products, Categories) và bảng transaction (Orders, Payments).
- Sử dụng **Soft Delete** (IsDeleted) cho Products để bảo toàn dữ liệu liên kết.
- ShipperLocations thiết kế append-only, hỗ trợ truy vấn nhanh theo CreatedAt DESC.

### 5.4 Trải nghiệm người dùng (UX/UI)
- Giao diện mobile-first, tối ưu cho thiết bị Android.
- Buttons và font chữ **to, rõ ràng**, dễ thao tác.
- **Ngôn ngữ tích cực**, không dùng ngôn từ gây miệt thị ngoại cỡ (body-shaming). Sử dụng: "Tự tin với phong cách của bạn", "Thoải mái trong mọi kích cỡ", thay vì "béo", "mập",...
- Hiển thị thông số size dưới dạng **trực quan** (biểu đồ, hình minh họa) thay vì chỉ số thuần.

### 5.5 Độ tin cậy (Reliability)
- Đảm bảo tính toàn vẹn dữ liệu: Foreign Keys, CHECK constraints, UNIQUE constraints.
- Snapshot thông tin sản phẩm vào OrderItems tại thời điểm mua (không bị ảnh hưởng khi sản phẩm thay đổi sau đó).
- Snapshot địa chỉ giao hàng vào Orders (không bị ảnh hưởng khi khách sửa UserAddresses).
